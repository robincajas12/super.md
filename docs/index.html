Your Markdown files run code when read

Virtual FS Node mounts a folder as a FUSE filesystem. When an AI agent reads a .md file, the system intercepts the syscall, executes embedded code blocks against the real project state, and returns live diagnostics — turning static checklists into self-correcting agent loops.

How it works

From Markdown to
executable constraint

Three things happen when an agent reads a mounted file: FUSE intercepts the read, processor.js finds and executes code blocks, and the output replaces the block inline — all before the bytes reach the caller.

01
Input
Write the checklist in Markdown
```run-node const fs = require('fs') const src = fs.readFileSync('src/Button.tsx', 'utf8') const hasProps = src.includes('interface ButtonProps') const hasTest = fs.existsSync('tests/Button.test.tsx') console.log(hasProps ? '- [x] Props typed' : '- [ ] Props typed') console.log(hasTest ? '- [x] Test file' : '- [ ] Test file') ```
Result
Plain text, no build step
The blocks only execute at read time. Edit the file like any Markdown document. Blocks run in WORKING_DIR — scoped to your project.

Executable blocks

Three block types,
one goal

Embed execution directly in Markdown using fenced code blocks with special tags. Each type runs in a configurable working directory — scoped to your project.

run-node

JavaScript

Executes Node.js code via stdin. Ideal for complex validation logic, file content checks, and structured diagnostics with the full power of npm.

```run-node const fs = require('fs') const pkg = JSON.parse( fs.readFileSync('package.json')) const hasLint = 'eslint' in (pkg.devDependencies||{}) const hasTypes = fs.existsSync('tsconfig.json') console.log(hasLint ? '[x] ESLint' : '[ ] ESLint') console.log(hasTypes ? '[x] TypeScript' : '[ ] TypeScript') ```
run

Shell command

Runs any shell command inside WORKING_DIR. Great for compilation checks, directory listings, and build validation with existing tools.

```run npm run lint -- --max-warnings 0 npm test -- --coverage --passWithNoTests docker build -t app:latest . 2>&1 | tail -3 ```
script

External script

Invokes a script stored in SCRIPTS_DIR. Keeps complex logic out of Markdown and reusable across multiple task files.

```script ./check-env-vars.sh ./verify-db-migrations.sh ./run-integration-tests.sh ```

Use cases

Where executable
docs help most

Any workflow where an AI agent needs to verify its own output is a candidate. The pattern generalizes across education, development, and operations.

Education

Auto-graded assignments

A teacher writes a checklist once. Students' agents read it, see exactly what's missing, fix the code, and re-read until all items pass. The teacher only reviews completed work — never partial attempts.

AI development

Self-correcting code agents

Instead of prompting an agent with vague goals, give it a mounted checklist. The agent verifies its own output at every step — structured constraints, not ambiguous instructions.

Team workflows

Living task documentation

Task docs that update themselves. A status.md file always shows the real project state because it checks actual files on every read — impossible to become stale.

CI / DevOps

Readable pipeline checks

Define pipeline constraints in plain Markdown. Any tool that can read a file can run your checks — no YAML DSL required, no build system lock-in, no custom runner to maintain.
